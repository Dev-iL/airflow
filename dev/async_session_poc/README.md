# Async-Session Execution-API POC

Synthetic A/B benchmark comparing async vs sync SQLAlchemy session usage in
the Airflow Execution API. The pedagogical question is whether converting a
FastAPI route from `def` + `SessionDep` to `async def` + `AsyncSessionDep`
relieves the AnyIO threadpool ceiling under concurrent DB-bound load.

The analysis and tables live in [`RESULTS.md`](RESULTS.md). This README is
the **operator runbook** for re-running the sweep.

## What is in this directory

| File | Purpose |
|------|---------|
| `seed.py` | Idempotently inserts 1000 `bench_var_*` Variables for `/var_count`. |
| `run_bench.py` | Pure-`httpx` concurrency-sweep harness. CLI; writes one CSV row per cell. |
| `run_all.sh` | Orchestrator. Prechecks asyncpg, emits `env_for_server.sh`, seeds, sweeps. |
| `RESULTS.md` | Tables + analysis + the AC-5.4 Anomalies section. |
| `results.csv` | 24-cell sweep output written by the orchestrator. |
| `diagnostics/` | Raw `pg_stat_activity` 1 Hz polling log captured during the sweep. |

## Prerequisites

- A breeze container running with `--backend postgres` (the bench routes use
  `pg_sleep`, asyncpg is the async driver).
- The Execution API server reachable on `localhost:8080` (default; override
  via `BENCH_TARGET_BASE_URL`).
- `asyncpg` importable inside breeze (the orchestrator prechecks this — see
  `R-7` / `INV-G18` in the manifest).
- The benchmark gate env var set on the API server process:
  `AIRFLOW__BENCHMARK__ENABLE_ASYNC_SESSION_ROUTES=true` (exact string).

## Re-running the sweep

```bash
# 1. Bring up postgres + airflow with the calibrated env vars
source dev/async_session_poc/env_for_server.sh  # generated by run_all.sh
breeze start-airflow --backend postgres

# 2. From the host, run the orchestrator
bash dev/async_session_poc/run_all.sh
```

The orchestrator generates `env_for_server.sh` with the canonical pool
calibration (`AIRFLOW__DATABASE__SQL_ALCHEMY_POOL_SIZE=40` and
`AIRFLOW__DATABASE__SQL_ALCHEMY_MAX_OVERFLOW=560`, combined ≥ 500 per
INV-G17, with ~50% headroom over the c=400 concurrency cap). On a clean
checkout, run `run_all.sh` once to emit `env_for_server.sh`, then source
that file in the breeze-launch shell.

## Tunable knobs (env-var overrides)

| Variable | Default | What it controls |
|----------|---------|------------------|
| `BENCH_TARGET_BASE_URL` | `http://localhost:8080` | Where the harness aims. |
| `BENCH_DURATION` | `30` | Seconds of measurement per cell. |
| `BENCH_WARMUP` | `5` | Seconds discarded before each cell's aggregates start (PG-12). |
| `BENCH_SLEEP_MS` | `500` | `pg_sleep` duration for the `/sleep` pair. |
| `BENCH_CONCURRENCIES` | `50 100 200 400` | Space-separated list of concurrency levels swept. |
| `AIRFLOW__DATABASE__SQL_ALCHEMY_POOL_SIZE` | `40` | Sync + async DB pool size (both engines share the key). |
| `AIRFLOW__DATABASE__SQL_ALCHEMY_MAX_OVERFLOW` | `560` | Overflow allowance. Sum with pool size must be ≥ 500 (INV-G17). |

To re-run a single cell or sub-grid:

```bash
BENCH_CONCURRENCIES="200" BENCH_DURATION=10 bash dev/async_session_poc/run_all.sh
```

## Re-running just the harness against a single URL

The harness is independent of the orchestrator:

```bash
uv run --project airflow-core python dev/async_session_poc/run_bench.py \
    --url http://localhost:28080/execution/__bench/sleep/async?ms=500 \
    --concurrency 200 --duration 30 --warmup 5 \
    --route sleep_500ms --mode async \
    --output dev/async_session_poc/results.csv
```

With `--route` and `--mode` omitted (self-test shape), the CSV header drops
the label columns and starts with `rps,…` — this matches the manifest's
AC-3.3 expected self-test format.

## Captured diagnostics

`diagnostics/pg_stat_activity.log` is a 1 Hz dump of
`pg_stat_activity.state` counts collected for the full duration of the
last sweep. Format: `timestamp|active_count|total_count`. Used by
`RESULTS.md`'s `## Anomalies` section to rule out DB-pool saturation
(R-6) — see the histogram and peak-active analysis there.

## Where this is *not* trying to go

- Not a PR. Not a newsfragment. Not a provider change. POC only.
- Not measuring Triggerer or Worker throughput — those components do not
  hit the metadata DB directly in 3.x. See the disclaimer paragraph in
  `RESULTS.md`.
- Not generalizing past one converted route (`get_variable_keys`) and one
  synthetic A/B pair. Multi-route migrations are a follow-up.
